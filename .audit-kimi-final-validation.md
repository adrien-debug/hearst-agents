## 🔧 ARCHITECTURE CORRIGÉE

### Flux de Persistence des Runs (Nouveau)

```
1. Client appelle /api/orchestrate
   ↓
2. orchestrate-v2.ts génère:
   - runId (format v2_timestamp_random)
   - dbRunId (UUID pour PostgreSQL)
   ↓
3. saveRun() persiste immédiatement:
   - id: dbRunId (UUID)
   - metadata: { runId, tenantId, workspaceId, ... }
   - status: "running"
   ↓
4. Exécution du backend (responses/assistants)
   ↓
5. À la completion:
   - updateRunMetrics(dbRunId, { tokens, cost, latency })
   - updateRun(dbRunId, { status: "completed", backend, completedAt })
   ↓
6. En cas d'erreur:
   - updateRun(dbRunId, { status: "failed", completedAt })
```

### Avantages de la nouvelle architecture

| Aspect | Avant | Après |
|--------|-------|-------|
| ID utilisé | runId v2 (pas UUID) | dbRunId UUID pour DB, runId v2 pour API |
| Persistence métriques | ❌ Échouait (UUID invalide) | ✅ Fonctionne (vrai UUID) |
| Corrélation | ❌ Impossible | ✅ runId stocké dans metadata |
| API Response | ❌ ID interne exposé | ✅ runId v2 cohérent avec events |

## ⚠️ POINTS DE ATTENTION

### 1. Assets - Double Source (Accepté)

| Source | Type | Persistence |
|--------|------|-------------|
| `POST /api/v2/assets` | In-memory Map | ❌ Volatile (restart) |
| Right Panel `assets` | Supabase DB | ✅ Persistant |

**Explication:** Les assets créés via l'API manuelle vont dans le Map (perdus au restart). Les assets créés par les runs vont dans Supabase (persistés). C'est temporairement acceptable.

### 2. OAuth Google - Non lié aux corrections

Le problème `invalid_client` est **indépendant** des corrections de métriques. Il faut vérifier:
- Client Secret dans Google Cloud Console
- OAuth Consent Screen configuration

### 3. Runs Historiques

Les 50 runs existants n'auront **pas** de métriques rétroactivement. Seuls les nouveaux runs (après redémarrage) auront:
- `metrics.tokensIn/Out`
- `metrics.costUsd`
- `metrics.latencyMs`
