001 Pipeline globale — Chat → Tool → Stage

Le système nerveux central. Un message user (texte ou voix) doit traverser cette pipeline pour déclencher un Stage spécialisé. Le tronçon "AI Pipeline → New Tools" n'est pas câblé — l'AI ne connaît que Composio + Google natif. C'est le gap structurel #1.

002 Les 7 signatures, par couche

Statut couche par couche : déclenchement (chat / hotkey / scheduler), tool handler dans l'AI pipeline, worker / route exécuteur, provider externe, Stage UI, et le branchement automatique chat→stage quand la signature aboutit.

004 Gaps qui empêchent la vision

P0 critique AI pipeline ne connaît pas les nouveaux tools
generate_image, browse_web, generate_video, parse_document, execute_code, start_meeting_bot, start_simulation, ingest_kg, query_kg sont déclarés dans lib/tools/registry.ts mais zéro handler dans lib/tools/handlers/ et zéro déclaration dans la liste aiTools de ai-pipeline.ts. Donc le chat ne peut PAS les déclencher. Le user doit naviguer manuellement dans chaque Stage.
Effort : ~1.5j · Impact : 6 signatures sur 7 deviennent triggerables depuis le chat
P0 critique Pas d'event stage_request → pas d'auto-switch
Un commentaire dans page.tsx:169 mentionne « tools côté backend qui SSE-broadcast un stage_request event » — mais le bus event n'a aucun émetteur ni listener pour ce type. Conséquence : même si un tool s'exécutait via le chat, l'utilisateur ne serait pas téléporté sur le Stage approprié.
Effort : ~0.5j · Impact : auto-switch chat → stage devient réel
P0 critique Voix : function calling absent + multi-sessions catastrophe
(a) Sig 6 livrée comme conversation pure : aucun bridge delegate_to_chat vers l'orchestrateur. La voix promet, mais ne fait aucune action backend.
(b) ✅ FIX commit d0bb955 (29/04 15:37) — VoicePulse monté au root layout via VoiceMount, singleton guards activePc + isStarting, reset() préserve voiceActive, bouton Couper avec fallback cockpit. Voix opérationnelle, 1 session par activation.
(a) Voice FC reste à faire — Effort : ~1j (Session R rédigée) · Impact : voix exécute pour de vrai
P1 Sessions meeting / browser / realtime sans ownership DB
Les routes GET /api/v2/meetings/[id] et /api/v2/browser/[id] n'ont pas de table d'ownership. Si user B connaît l'ID d'une session de user A, il y accède. Théorique mais réel — bloquant avant d'ouvrir l'app à plusieurs users.
Effort : ~0.5j · Impact : sécu prod multi-tenant
P1 Stagehand absent dans browser-task
BrowserStage affiche le live viewer Browserbase, mais l'agent ne navigue pas réellement (pas de Stagehand). Co-Browsing aujourd'hui = simple iframe live d'une session vide. Pas de vraie automatisation.
Effort : ~1j · Impact : Sig 3 devient ce que la vision décrit
P1 Letta / mémoire long terme absente
KG actuel = extraction d'entités + viz Cytoscape. Pas de vector search, pas de consolidation, pas de raisonnement long terme. Letta self-hosted (Docker) à brancher pour atteindre la vision « la 50e conversation, l'agent te connaît ».
Effort : ~2j · Impact : KG passe d'outil de visualisation à mémoire vivante
P2 Apollo / PDL B2B intel non branché
Clés présentes en env, mais pas de provider, pas de tool, pas d'intégration au chat. Phase 2.6 du plan.
Effort : ~0.5j · Impact : trouver/enrichir des contacts B2B depuis le chat
P2 Hydration mismatch + ergonomies UI mineures
Quelques ActionChip morts dans ChatMessages, briefing matinal sans CTA visible quand non généré, transcript voix doublonné centre + rail (résolu par Session S).
Effort : ~0.5j · Impact : polish
